mt — CI/CD ManualGitHub Actions workflow reference

Overview #

mt uses a single GitHub Actions workflow file, .github/workflows/ci.yml, that runs automatically on every push and every pull request. It executes two parallel jobs — test and build — giving fast feedback on both code correctness and compilation.

The workflow is intentionally minimal: no deployment, no release automation, no matrix builds. Its only job is to catch regressions before they land on main.

.github/
└── workflows/
    └── ci.yml   ← the only workflow file

Triggers #

The workflow runs on two event types:

on:
  push:
    branches: ["**"]
  pull_request:

On push — every branch

branches: ["**"] matches every branch name without exception. This means CI runs the moment you push any commit anywhere — not just to main or protected branches.

Why: Developers working on feature branches get immediate feedback without waiting for a PR. A broken build on a long-running branch is caught before it diverges further from a working state. The cost is a few extra CI minutes per push, which is worthwhile for a small repo.

On pull request

The bare pull_request: trigger fires on the following PR events by default: opened, synchronize (new commit pushed to a PR branch), and reopened. No branch filter is applied, so it covers PRs targeting any base branch.

Why: GitHub uses the PR check run results to gate merges. Branch protection rules can require the test and build checks to pass before a PR is mergeable. The pull_request trigger is what populates those required status checks.

Push + PR: is CI run twice? When you push to a branch that has an open PR, both triggers fire — GitHub runs CI for the push event and again for the PR synchronize event. This is normal and harmless; it means the PR status check reflects the latest commit regardless of whether the PR was open at push time.

Jobs #

The workflow defines two independent jobs that run in parallel on separate runners:

JobRunnerStepsPurpose
test ubuntu-latest checkout → setup-go → go vet → go test Verify code correctness: static analysis and test suite
build ubuntu-latest checkout → setup-go → go build Verify the binary compiles cleanly from a clean checkout

test job

Runs static analysis with go vet followed by the full test suite with go test. Vet runs first because it is faster and catches issues that would otherwise produce confusing test failures (e.g. a misformatted Printf call or an unreachable code path).

build job

Compiles the final binary with go build -o mt .. This is the same command used locally (make build) so it confirms the exact artifact that would be distributed actually compiles.

↓
2
actions/setup-go@v5
Same Go version as the test job (both read go.mod)
↓
3
go build -o mt .
Compiles the root package to a binary named mt; fails if any file has a compile error

Why two separate jobs?

Running test and build as independent jobs (not sequential steps in one job) has two concrete benefits:

  1. Parallelism. Both jobs start at the same time on separate runners. Total wall-clock CI time is determined by whichever finishes last, not their sum. On a cold runner this saves roughly 30–60 seconds per run.
  2. Distinct failure signals. If test fails and build passes, you know the tests themselves are broken, not compilation. If build fails but test passes, something in main.go or a non-test file has a compile error that tests did not exercise. The two signals are meaningfully different.
The build job does not run tests go build compiles but does not execute test files. A test that panics at runtime will be caught by the test job, not the build job. Both checks must be green for a commit to be considered passing.

Step reference #

actions/checkout@v4

Provided by GitHub. Clones the repository into /home/runner/work/mt/mt at the SHA that triggered the workflow. For push events this is the pushed commit; for pull requests it is the merge commit of the PR head into the base branch.

Why v4: Version 4 uses sparse checkouts by default and supports Git 2.18+ features. It is the current stable release and the one most Actions documentation refers to.

actions/setup-go@v5

- uses: actions/setup-go@v5
  with:
    go-version-file: go.mod
    cache: true

Installs the Go toolchain and configures the module cache. Two options are set:

OptionValueEffect
go-version-file go.mod Reads the go directive from go.mod to pick the exact toolchain version — currently Go 1.24.7. No hardcoded version string in the workflow file; the workflow automatically tracks go.mod changes.
cache true Caches the Go module download cache ($GOPATH/pkg/mod) keyed on go.sum. Subsequent runs that do not change dependencies restore modules from cache instead of re-downloading from the internet. See Dependency caching.

go vet ./...

- name: vet
  run: go vet ./...

go vet is the Go standard library's built-in static analyser. It runs a collection of checks that the compiler does not enforce but that almost always indicate bugs:

  • Printf format/argument mismatch — e.g. fmt.Sprintf("%d", "not an int")
  • Unreachable code after return/panic
  • Struct tag syntax errors — malformed yaml: tags would silently break config parsing
  • Copying of mutexes by value
  • Incorrect use of sync/atomic
  • Loop variable capture in goroutines (pre-Go 1.22)

Vet runs before tests because its checks are cheap (no execution) and a vet failure on a Sprintf call often manifests as a confusing test failure rather than a clear error.

The ./... pattern matches every package in the module, including test packages.

go test ./...

- name: test
  run: go test ./...

Compiles and runs every _test.go file in the module. ./... recursively descends into subdirectories, so this covers both internal/tui/ (the main package) and any future packages added under the module root.

Key behaviours of go test in this configuration:

  • Each package's tests run in a fresh binary — test failures in one package do not prevent other packages from running.
  • A non-zero exit from any test binary causes the overall step to fail.
  • Test binaries that panic are reported as failures, not crashes.
  • The race detector is not enabled here (no -race flag). The test suite can be run with go test -race ./... locally for deeper concurrency checking.
Coverage No coverage report is generated or uploaded by this workflow. Test coverage can be inspected locally with go test -cover ./... or go test -coverprofile=coverage.out ./... && go tool cover -html=coverage.out.

go build -o mt .

- name: build
  run: go build -o mt .

Compiles the module's root package (main) and all its transitive dependencies into a single binary named mt. The -o mt flag names the output file explicitly; without it Go would use the module name.

Why this step is separate from go test: go test ./... does compile all non-test packages to build the test binaries, so a compile error in a library package would surface there. However, go build . exercises the main package's linkage and any //go:embed directives independently, and it verifies the exact build command developers use locally matches CI.

The produced mt binary is not archived or uploaded — this step exists purely as a compilation gate. No artifact is retained after the job finishes.

Go version management #

The Go version used by CI is determined entirely by the go directive in go.mod:

go 1.24.7

Because both jobs set go-version-file: go.mod, updating the minimum Go version in go.mod automatically changes the version used in CI on the next push — no workflow file edit needed.

Minimum vs. exact version The go directive in go.mod sets the minimum language version required by the module. actions/setup-go interprets this as the exact version to install. If the directive says go 1.24.7, CI will install Go 1.24.7 specifically. To allow any patch release of 1.24, you would write go 1.24 (without a patch version).

To upgrade Go locally and in CI at the same time:

go mod edit -go 1.25.0
go mod tidy
git commit go.mod go.sum -m "chore: upgrade to Go 1.25.0"

Dependency caching #

With cache: true in actions/setup-go@v5, the action automatically saves and restores the Go module download cache ($GOPATH/pkg/mod/cache) between runs. The cache key is derived from the hash of go.sum.

ScenarioCache behaviour
No dependency changes since last run Full cache hit — modules restored in seconds, no network I/O
go.sum changed (new or updated dependency) Cache miss — modules re-downloaded; new cache entry saved for future runs
First run ever, or cache evicted by GitHub Cold download — takes 10–20 s for mt's small dependency tree

The mt module has a small dependency tree (Bubble Tea, Lipgloss, and their transitive deps). Even on a cold cache the download takes under 30 seconds. The cache primarily benefits push-heavy workflows where CI runs many times per day.

Cache is per-OS, per-architecture GitHub Actions caches are scoped to the OS and architecture of the runner. Both jobs use ubuntu-latest (x86-64 Linux), so they share a cache. If you add a macOS or ARM job in the future, it will have a separate cache entry.

Reading CI results #

After a push or PR event, CI results appear in two places:

On a pull request

The PR page shows a status checks section near the merge button. Each job (test and build) appears as a separate check. Click Details next to any check to open the full job log.

On a commit or branch

Navigate to Actions in the repository sidebar, or click the coloured dot next to a commit SHA. Green circle = all jobs passed. Red X = at least one job failed. Yellow dot = jobs in progress.

Interpreting failures

Failing jobFailing stepLikely cause
test vet Static analysis issue: mismatched Printf arguments, struct tag typo, mutex copy. Fix with go vet ./... locally.
test test A test assertion failed, or a package failed to compile for test. Run go test ./... locally; check for panics in test output.
build build The main package or one of its dependencies has a compile error not caught by tests. Run go build -o /dev/null . locally.
Either setup-go The Go version in go.mod is not yet available on GitHub-hosted runners (rare). Check the actions/setup-go release notes.

Extending the workflow #

Common additions to consider as the project grows:

Enable the race detector

Replace the test step with:

- name: test
  run: go test -race ./...

The race detector adds ~2× overhead and requires CGO. It is off by default to keep CI fast; enable it if concurrency bugs become a concern.

Add a coverage upload

- name: test
  run: go test -coverprofile=coverage.out ./...

- name: upload coverage
  uses: codecov/codecov-action@v4
  with:
    files: coverage.out

Add a release job

To build release binaries for multiple platforms when a tag is pushed:

on:
  push:
    tags: ["v*"]

jobs:
  release:
    strategy:
      matrix:
        goos: [linux, darwin]
        goarch: [amd64, arm64]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-go@v5
        with:
          go-version-file: go.mod
          cache: true
      - name: build
        env:
          GOOS: ${{ matrix.goos }}
          GOARCH: ${{ matrix.goarch }}
        run: go build -o mt-${{ matrix.goos }}-${{ matrix.goarch }} .

Restrict CI to fewer branches

If the current all-branch trigger produces too many CI runs, narrow it:

on:
  push:
    branches: [main, "release/**"]
  pull_request:

Feature branches then only get CI when a PR is opened against them.

Secrets in workflows The current workflow uses no secrets. If you add steps that require credentials (registry login, deployment keys), always store them in repository secrets (Settings → Secrets and variables → Actions) and reference them as ${{ secrets.MY_SECRET }}. Never hardcode credentials in workflow YAML.

Local parity #

The CI steps map exactly to the Makefile targets. Running make all locally executes the same commands in the same order:

CI stepLocal equivalentMake target
go vet ./...go vet ./...make vet
go test ./...go test ./...make test
go build -o mt .go build -o mt .make build

Running make all (or make vet test build) before pushing gives you high confidence that CI will pass. The only difference between local and CI is the operating system — CI runs on ubuntu-latest, while most contributors develop on macOS. Both are supported Go platforms with no platform-specific code paths in mt.

Tip: run CI checks before pushing make all runs vet, test, and build in sequence and exits non-zero on the first failure — the same behaviour as CI. Add it to a pre-push git hook to catch failures before they reach GitHub.
